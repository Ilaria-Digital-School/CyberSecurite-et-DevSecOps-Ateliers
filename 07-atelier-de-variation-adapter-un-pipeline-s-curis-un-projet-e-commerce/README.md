# Atelier de variation : adapter un pipeline sécurisé à un projet e-commerce

**Rappel théorique** Un pipeline CI/CD (Intégration Continue / Déploiement Continu) est une chaîne automatisée qui va du code source jusqu'à la mise en production. Dans une approche DevSecOps, la sécurité n'est pas ajoutée à la fin : elle est intégrée à chaque étape. On parle de "Shift Left" — déplacer les contrôles de sécurité le plus tôt possible dans le cycle de développement. Les trois piliers à retenir : le SAST (analyse statique du code source avant exécution), le DAST (analyse dynamique de l'application en cours d'exécution) et le SCA (analyse des dépendances tierces pour détecter les bibliothèques vulnérables). À ces outils s'ajoutent des contrôles sur les secrets (pas de mots de passe en dur dans le code), la sécurité des conteneurs et les tests d'intégration. Erreur fréquente des débutants : croire qu'un pipeline sécurisé est identique pour tous les projets. En réalité, le contexte métier impose des priorités différentes. Un site e-commerce traite des données bancaires, des sessions utilisateur et des flux de commandes — ces éléments doivent se refléter dans les contrôles de sécurité choisis et leur positionnement dans le pipeline. Mini-exemple : dans un pipeline générique, on place un SAST après le commit. Pour un e-commerce, on ajoutera obligatoirement une analyse des dépendances (SCA) sur les bibliothèques de paiement, et un DAST ciblant les formulaires de connexion et le tunnel de commande. --- **Énoncé de l'atelier** Vous avez travaillé lors des ateliers précédents sur un pipeline CI/CD simplifié générique.

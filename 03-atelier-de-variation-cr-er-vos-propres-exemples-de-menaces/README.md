# Atelier de variation : créer vos propres exemples de menaces

**Rappel théorique** Avant de plonger dans l'atelier, voici un rappel des concepts essentiels abordés dans les sessions précédentes. Une **menace** est tout événement ou acteur susceptible d'exploiter une **vulnérabilité** (faille ou faiblesse dans un système) pour causer un préjudice. Le **risque** représente la probabilité qu'une menace se concrétise et l'impact que cela entraîne. Ces notions sont indissociables des trois piliers CIA : **Confidentialité** (seules les personnes autorisées accèdent aux données), **Intégrité** (les données ne sont pas altérées sans autorisation) et **Disponibilité** (les services restent accessibles quand on en a besoin). Une erreur fréquente chez les débutants est de confondre la menace et la vulnérabilité. La vulnérabilité est la porte entrouverte ; la menace est le cambrioleur qui pourrait la pousser. Une autre erreur courante est d'oublier de préciser quel pilier CIA est impacté, ce qui rend l'analyse incomplète et inexploitable professionnellement. Mini-exemple : Un serveur web expose une version obsolète d'OpenSSL (vulnérabilité). Un attaquant exploite cette faille pour intercepter des communications chiffrées (menace). Les données clients sont lues sans autorisation → atteinte à la **Confidentialité**. Le risque est élevé si le serveur traite des données bancaires. Gardez en tête ce modèle mental : **Contexte → Vulnérabilité → Menace → Pilier CIA impacté → Niveau de risque**.

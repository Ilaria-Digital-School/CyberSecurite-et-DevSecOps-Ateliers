# Atelier : premières manipulations Git (clone, commit)

**Rappel théorique** Git est un système de gestion de versions distribué. Chaque développeur ou ingénieur DevSecOps dispose d'une copie complète de l'historique du projet (le dépôt ou 'repository'). Trois concepts fondamentaux à maîtriser immédiatement : 1. Le **Working Directory** : votre dossier de travail local, là où vous éditez vos fichiers. 2. Le **Staging Area (Index)** : une zone tampon où vous sélectionnez précisément quels changements seront inclus dans votre prochain commit. Commande clé : `git add`. 3. Le **Repository local** : la base de données immuable qui stocke l'historique. Commande clé : `git commit`. Erreur fréquente des débutants : faire `git add .` sans vérifier ce qu'ils ajoutent, et commettre des fichiers sensibles (clés API, mots de passe, fichiers `.env`). Dans un contexte DevSecOps, cela représente une faille de sécurité critique — ces secrets se retrouvent dans l'historique pour toujours, même après suppression du fichier. Mini-exemple mental : imaginez Git comme un journal de bord de bord maritime. Chaque commit est une entrée datée et signée. Si quelqu'un modifie une entrée, cela se voit immédiatement. C'est exactement pour cette raison que Git est le premier maillon de la chaîne de livraison sécurisée (CI/CD) : chaque modification de code est tracée, attribuée et vérifiable. --- **Énoncé de l'atelier** Vous venez de découvrir Git comme socle du pipeline DevSecOps dans le chapitre précédent.

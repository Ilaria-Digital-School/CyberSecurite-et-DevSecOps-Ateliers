# Atelier : simuler un secret committé et le corriger

## Rappel théorique Un 'secret' dans le contexte du développement logiciel désigne toute donnée sensible qui ne doit jamais apparaître dans un dépôt de code : mots de passe, clés API, tokens d'authentification, chaînes de connexion à une base de données, clés privées SSH ou TLS. Le problème fondamental est que Git conserve l'intégralité de l'historique des commits. Même si vous supprimez un fichier contenant un secret dans un commit ultérieur, ce secret reste accessible dans l'historique via `git log`, `git show` ou `git checkout` sur l'ancien commit. C'est ce qu'on appelle un secret 'committé'. Erreurs fréquentes : croire qu'un `git rm` ou un `git commit --amend` suffit à effacer définitivement un secret, ou penser qu'un dépôt privé est entièrement sûr (les dépôts privés peuvent être clonés, forkés, exposés par erreur). Le modèle mental à retenir : **Git = mémoire permanente**. Chaque commit est une empreinte immuable. Mini-exemple : un développeur ajoute `API_KEY=sk-abc123` dans un fichier `config.py`, le committe, puis réalise son erreur et supprime la ligne dans un nouveau commit. La clé est toujours visible avec `git show HEAD~1:config.py`. La correction réelle nécessite une réécriture de l'historique ET la révocation immédiate de la clé compromise. --- ## Énoncé de l'atelier Dans cet atelier, vous allez reproduire le scénario le plus courant et le plus dangereux qui soit dans le quotidien d'un développeur ou d'un ingénieur DevSecOps : committer accidentellement un secret dans un dépôt Git, puis le détecter et le corriger de manière professionnelle.

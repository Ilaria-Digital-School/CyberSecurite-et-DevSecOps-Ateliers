# Atelier : extraire les informations clés du rapport

## Rappel théorique Un rapport de vulnérabilités est un document structuré qui consigne les failles découvertes lors d'un audit ou d'un pentest. Il ne s'agit pas d'une simple liste de bugs : chaque entrée contient des informations précises qui permettent à une équipe de prioriser et de corriger efficacement. Les éléments fondamentaux d'une entrée de rapport sont : l'identifiant de la vulnérabilité (souvent un CVE ou un identifiant interne), la sévérité (Critical, High, Medium, Low, Informational), le score CVSS le cas échéant, la description technique, la preuve (proof of concept, capture d'écran, payload), l'impact métier potentiel, et la recommandation de remédiation. Le modèle mental à retenir : lire un rapport, c'est répondre à trois questions essentielles — **Quoi ?** (quelle vulnérabilité), **Où ?** (quel composant, quelle URL, quel endpoint), **Pourquoi c'est grave ?** (quel impact sur la CIA : Confidentialité, Intégrité, Disponibilité). La classification OWASP Top 10 vue précédemment sert ici de référentiel pour catégoriser les failles rencontrées. Erreur fréquente des débutants : se concentrer uniquement sur les vulnérabilités critiques et ignorer les Medium ou Low, alors que des failles de faible sévérité combinées entre elles peuvent constituer un vecteur d'attaque sérieux (chaining). Mini-exemple : une faille "Information Disclosure" (Low) qui expose la version d'un framework combinée à une CVE publique sur cette version devient immédiatement exploitable. --- ## Énoncé de l'atelier Dans cet atelier, vous allez travailler à partir du rapport fictif de vulnérabilités lu lors de l'activité précédente ("Lecture d'un rapport fictif de vulnérabilités : structure et terminologie")....

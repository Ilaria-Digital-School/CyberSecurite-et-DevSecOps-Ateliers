# Atelier : placer les tests sécurité dans un workflow existant

**Rappel théorique** Un workflow de développement (ou pipeline CI/CD) est une chaîne d'étapes automatisées qui transforme du code source en application déployée. L'enjeu du DevSecOps est d'intégrer les contrôles de sécurité à chaque étape de cette chaîne, sans bloquer la productivité. On distingue trois grandes familles de tests sécurité : le SAST (Static Application Security Testing), qui analyse le code source sans l'exécuter et détecte les vulnérabilités dès l'écriture du code ; le DAST (Dynamic Application Security Testing), qui attaque l'application en cours d'exécution pour trouver des failles réelles ; et le SCA (Software Composition Analysis), qui vérifie les dépendances tierces et bibliothèques open source utilisées. Le principe Shift-Left signifie déplacer ces tests le plus tôt possible dans le cycle, idéalement avant même le merge d'une branche. L'erreur fréquente des équipes débutantes est de ne placer les tests de sécurité qu'en fin de pipeline (juste avant la production), ce qui rend les corrections coûteuses et tardives. Mini-exemple concret : un développeur pousse du code Python contenant une dépendance vulnérable (CVE connue). Un outil SCA placé dès la PR (Pull Request) bloque le merge et remonte l'alerte immédiatement, évitant que la faille n'atteigne la production. La règle d'or : chaque type de test a sa place naturelle dans le pipeline, et les placer intelligemment est un acte d'architecture autant que de sécurité. --- **Énoncé de l'atelier** Dans cet atelier, vous allez analyser un workflow CI/CD existant (fictif mais réaliste), identifier ses lacunes en matière de sécurité, puis proposer et documenter l'intégration...

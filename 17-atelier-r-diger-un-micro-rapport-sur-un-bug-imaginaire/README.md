# Atelier : Rédiger un micro-rapport sur un bug imaginaire

**Rappel théorique** Un rapport de vulnérabilités est un document structuré qui permet de communiquer une faille de sécurité de manière claire, reproductible et exploitable par les équipes de développement et de management. Il ne s'agit pas simplement de dire « il y a un bug » : un bon rapport permet à un lecteur non-technique de comprendre l'impact, et à un développeur de reproduire et corriger la faille sans ambiguïté. Les composantes fondamentales d'un rapport de vulnérabilité sont : l'identifiant ou titre (clair et précis), la description de la vulnérabilité, la sévérité (critique, haute, moyenne, faible – souvent basée sur CVSS ou une évaluation CIA), les étapes de reproduction (preuve de concept), l'impact potentiel (lien avec la triade CIA : Confidentialité, Intégrité, Disponibilité), les références (OWASP Top 10, CVE…) et les recommandations de correction. Erreurs fréquentes des débutants : décrire la faille sans étapes de reproduction claires, négliger l'impact réel sur le métier, confondre sévérité et priorité de correction, ou utiliser un jargon non défini. Mini-exemple : « Titre : Injection SQL sur le formulaire de connexion. Sévérité : Critique. Description : Le champ « username » n'est pas filtré, permettant d'injecter des commandes SQL. Reproduction : saisir `' OR '1'='1` dans le champ. Impact : accès non autorisé à tous les comptes (Confidentialité compromise). Correction recommandée : utiliser des requêtes préparées. » Ce rappel fait le pont avec les activités précédentes : la lecture du rapport fictif, l'extraction d'informations clés et la terminologie OWASP Top 10 que vous avez...

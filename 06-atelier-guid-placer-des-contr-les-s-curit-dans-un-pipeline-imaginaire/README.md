# Atelier guidé : placer des contrôles sécurité dans un pipeline imaginaire

**Rappel théorique** Un pipeline CI/CD (Continuous Integration / Continuous Delivery) est une chaîne automatisée qui va du code source jusqu'au déploiement en production. Dans une approche DevSecOps, la sécurité n'est pas ajoutée à la fin : elle est intégrée à chaque étape du pipeline, selon le principe du 'Shift Left' (déplacer la sécurité vers la gauche, c'est-à-dire le plus tôt possible dans le cycle de développement). Les principaux contrôles de sécurité que l'on peut placer dans un pipeline sont : le SAST (Static Application Security Testing), qui analyse le code source sans l'exécuter pour détecter des vulnérabilités ; le SCA (Software Composition Analysis), qui vérifie si les bibliothèques tierces utilisées contiennent des failles connues (CVE) ; le DAST (Dynamic Application Security Testing), qui attaque l'application en cours d'exécution pour trouver des failles ; la vérification des secrets exposés (tokens, mots de passe dans le code) ; et le scan des images conteneurs. Erreur fréquente des débutants : penser que la sécurité ne concerne que le déploiement final. En réalité, une faille introduite en phase de développement coûte 10 fois plus cher à corriger si elle est détectée en production. Mini-exemple : si un développeur pousse du code contenant un mot de passe en dur, un outil de détection de secrets (ex. GitLeaks) placé dès l'étape de commit peut bloquer immédiatement cette mauvaise pratique avant qu'elle n'atteigne le dépôt distant. --- **Énoncé de l'atelier** Dans cet atelier, vous allez travailler sur un pipeline CI/CD imaginaire correspondant à une application web fictive...

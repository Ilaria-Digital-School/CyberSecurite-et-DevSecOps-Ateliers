# Atelier : configurer une règle pre-commit basique

**Rappel théorique** Un hook Git est un script qui s'exécute automatiquement à un moment précis du cycle de vie d'un dépôt Git : avant un commit, après un push, etc. Le hook pre-commit est déclenché juste avant que Git enregistre le commit. C'est le moment idéal pour effectuer des vérifications automatiques : syntaxe, formatage, et surtout détection de secrets (mots de passe, clés API, tokens) qui ne devraient jamais se retrouver dans l'historique. Le framework open-source `pre-commit` (https://pre-commit.com) simplifie la gestion de ces hooks : il suffit de déclarer les règles souhaitées dans un fichier `.pre-commit-config.yaml` à la racine du dépôt, puis d'installer les hooks avec une seule commande. Les règles (ou « hooks ») proviennent de dépôts distants standardisés, par exemple `detect-secrets` ou `gitleaks`. Erreur fréquente : croire que le pre-commit remplace un scan côté CI/CD. Non — il constitue le premier rempart, côté développeur, mais il reste contournable localement (option `--no-verify`). C'est pourquoi il s'inscrit dans une défense en profondeur, aux côtés des politiques de branche et des pipelines SAST vus dans les chapitres précédents. Mini-exemple : un développeur ajoute par mégarde `API_KEY=sk-1234abcd` dans un fichier de config. Sans pre-commit, ce secret est committé et potentiellement poussé. Avec une règle `detect-secrets` active, le commit est bloqué immédiatement et le développeur reçoit un message explicite. --- **Énoncé de l'atelier** Vous avez exploré les politiques de branche et simulé la correction d'un secret committé par erreur.
